1. Only call the api for checkAuth when:

2. Difference between localStorage and httpOnlyCookie auth > DONE
    - localStorage asks Authorization: Bearer {{token}} in the header and stored in the localStorage which is
    prone to XSS since it can be accessed through JS 
    - httpOnlyCookie only asks withCredentials every requests so the token is not prone to attack and not
    accessible through JS
3. Persistency of auth data upon refresh/manual url > DONE
    - With the use of httpOnlyCookie, the data in the backend is persistent. However in the frontend, its not possible with that only.
    Storing the userData in the localStorage will help upon persistency of user data in the frontend.
    HOWEVER! local/session storage are not advisable for storing user data (other data like cart etc is good), but there are packages 
    that will help with persistency. For vue: Vue + Pinia + pinia-plugin-persistedstate so the data is persistent without saving on localStorage
4. Difference between options and composition api > DONE 
    - options api is old type of vue syntax wherein it manually defines the life cycle in vue (from data > method etc) > only script no setup
    - composition api is the modern and best practice currently in vue syntax similar to react (with only script setup)

6. Naming convention of API endpoints. API endpoints must always be in NOUNS and not VERBS. So technically a brief description of what the request does
    - EX: Get all freelances - /freelances
        Get a user’s freelances - /users/{id}/freelances
    > FOR METHODS its the same! 
        > EX: /freelances > index();
        > Get freelances by user > getFreelancesByUser($userId)
    > FOR SPECIFIC EXAMPLE: 
        ✅ Use Plural Nouns (e.g., /freelances, /users)
        ✅ Use Subresources for Relationships (e.g., /users/{id}/freelances)
        ✅ Use Query Parameters for Filtering (e.g., /freelances?user_id=1)
        ✅ Avoid Verbs (since HTTP methods already describe the action)
7. Difference between Unit and feature tests
    > Technically Feature tests interacts with database while unit does not 
        > so does that mean that only basic features without any database interactions/http or api request will be able to be handle by
            unit test?
            > YES!
    > Unit test technically only tests the business logic (such as services,utility and helper classes)
        > Unit tests can also be from input/form for validation purposes
    > Feature tests are the one for real world simulation (log in, simulation of app, submitting forms etc)
    - PHPUnit is already available in the laravel and can be used easily, 
    > to create one, just call php artisan make:test Name (then add --unit for unit testing)
